# Letta + Doubao Manual

This document is the durable handoff for the standalone bundle. It records the design decisions, what was verified, what failed, and what to keep in mind when moving this directory somewhere else.

## Goal

Create a standalone directory that can run Letta against Doubao without depending on the rest of the Letta source repository.

## Current State

The current bundle works end to end with:

- Letta server in Docker
- Postgres + pgvector in Docker
- Doubao Ark as the LLM backend through Letta's OpenAI-compatible provider path
- Letta's built-in embedding backend for agent memory
- Two marimo notebook files that can also be executed as script-based smoke tests

## Directory Contents

- `compose.yaml`: standalone Docker Compose stack
- `init.sql`: required database bootstrap
- `.env.example`: sanitized configuration template
- `.env`: local runtime configuration
- `pyproject.toml`: notebook dependency manifest for `uv`
- `uv.lock`: dependency lockfile generated by `uv`
- `README.md`: short run instructions
- `MANUAL.md`: this longer decision log and runbook
- `notebooks/01_doubao_api_smoke.py`: direct Ark validation
- `notebooks/02_letta_e2e.py`: Letta end-to-end validation

## Core Decisions

### 1. Standalone Directory Instead Of Reusing The Repo Root

Decision:
- Package everything under one directory so it can be copied elsewhere unchanged.

Why:
- The repo checkout is useful for investigation, but it should not be a runtime requirement.
- The final objective was future reuse after moving to a clean directory.

Consequence:
- No source build is required.
- The standalone directory is the artifact to keep.

### 2. Docker Compose Over Dockerfile

Decision:
- Use Docker Compose with the official `letta/letta:latest` image instead of building from the repo Dockerfile.

Why:
- The user explicitly preferred Compose.
- The official image is enough for this setup.
- Avoiding a build removes unnecessary moving parts.

Consequence:
- The standalone bundle does not need the repo Dockerfile.
- There is currently no Docker build context in this bundle.

### 3. Minimal Services

Decision:
- Keep only:
  - Letta server
  - Postgres + pgvector

Why:
- Nginx is not required for local validation.
- Fewer services make the bundle easier to move and debug.

Consequence:
- Letta API is exposed directly on `8283`.
- Letta ADE/UI is exposed directly on `8083`.

### 4. Doubao Through Letta's OpenAI-Compatible Path

Decision:
- Configure Letta via:
  - `OPENAI_API_BASE=https://ark.cn-beijing.volces.com/api/v3`
  - `OPENAI_API_KEY=<ark key>`

Why:
- Letta already supports OpenAI-compatible providers through this path.
- Ark supports compatible `/models` and `/chat/completions` behavior.

Consequence:
- Letta discovers Doubao-backed models as `openai-proxy/<model-id>`.
- The tested model handle is `openai-proxy/doubao-seed-1-8-251228`.

### 5. Doubao For LLM, Not For Embeddings

Decision:
- Use Doubao for the agent LLM.
- Use `letta/letta-free` for embeddings.

Why:
- The Ark key successfully supported:
  - `GET /models`
  - `POST /chat/completions`
  - OpenAI-style tool calling
- The tested embedding requests did not work with the same key on the tested compatibility path.

What failed:
- Alias-style embedding model name: `doubao-embedding`
- Tested explicit IDs returned by `/models`, such as:
  - `doubao-embedding-text-240715`
  - `doubao-embedding-large-text-240915`

Observed result:
- Ark returned `InvalidEndpointOrModel.NotFound` for those embedding requests with this key.

Consequence:
- The current working end-to-end stack is not "pure Doubao" for embeddings.
- If pure Doubao embeddings are required later, that needs a separate investigation.

### 6. Keep `init.sql`

Decision:
- Keep `init.sql` in the standalone bundle.

Why:
- It is still necessary for this database layout.
- It does all of the following:
  - connects to the target database
  - creates a dedicated schema named after the DB
  - sets the database `search_path`
  - creates the `vector` extension in that schema
  - drops the default `public` schema

Consequence:
- `init.sql` is a main file and should be preserved.
- Removing it changes the DB initialization behavior and would likely create avoidable startup/runtime issues.

### 7. Use `uv` Instead Of `pip`

Decision:
- Use `uv` as the dependency manager for the notebooks.

Why:
- The user explicitly requested it.
- The main Letta repo already uses `uv`.
- `uv sync` + `uv.lock` gives a cleaner portable workflow than `pip install -r requirements.txt`.

Consequence:
- `pyproject.toml` and `uv.lock` are the source of truth for Python dependencies.
- The old `requirements.txt` was removed to avoid two competing workflows.

### 8. Add `.dockerignore`, But Treat It As Future-Proofing

Decision:
- Add a minimal `.dockerignore`.

Why:
- Today it has no effect because the standalone Compose file does not build an image.
- It becomes useful immediately if a future version adds a `build:` section or a `docker build` command.

Consequence:
- `.dockerignore` is harmless now.
- It mainly prevents accidental inclusion of `.env`, `.venv`, local data, and caches in any future build context.

## Verified Findings

These were actually tested, not assumed:

### Ark / Doubao Directly

Verified:
- `GET /models` works with the provided key.
- `POST /chat/completions` works with `doubao-seed-1-8-251228`.
- A tool-calling request produced an OpenAI-style `tool_calls` payload.

Direct smoke output confirmed:
- model discovery
- exact chat response
- exact function name in returned `tool_calls`

### Letta End To End

Verified:
- Letta container starts cleanly with the Ark compatibility base URL.
- Letta lists the Doubao model handle:
  - `openai-proxy/doubao-seed-1-8-251228`
- Letta lists the embedding handle:
  - `letta/letta-free`
- Letta can create an agent with those handles.
- Letta can run a message through that agent and return:
  - `Letta via Doubao ok`

## Runtime Details That Matter

### Model Handle Naming

Important:
- In Letta, the Doubao model handle is not `openai/doubao-seed-1-8-251228`.
- It is `openai-proxy/doubao-seed-1-8-251228`.

Reason:
- Letta uses `openai-proxy` for non-OpenAI providers routed through the OpenAI-compatible provider implementation.

### Full Model ID Matters

Important:
- Use the actual Ark model ID, not the display name.

Working:
- `doubao-seed-1-8-251228`

Do not assume:
- `doubao-seed-1-8` alone will behave the same way in Letta's discovered model list.

### Letta Model Routes Redirect

Observed:
- Letta's model listing route redirects to a trailing-slash URL.

Example:
- `GET /v1/models` redirected to `GET /v1/models/`

Consequence:
- The e2e notebook was updated to use `follow_redirects=True` and `/v1/models/`.

### Marimo Notebook Files Also Run As Scripts

Decision:
- The notebook files support both:
  - normal marimo UI usage
  - script-style smoke execution

Mechanism:
- Set `MARIMO_SMOKE_ONLY=1` and run the notebook with Python.

Why:
- This makes smoke tests automatable without opening the notebook UI.

## Fresh Start Runbook

If you move this directory somewhere else and want a clean start:

1. Copy the entire directory.
2. Keep or recreate `.env`.
3. If you want a truly fresh DB, remove `data/pgdata`.
4. Run:

```powershell
uv sync
docker compose up -d
```

5. Run the smoke notebooks:

```powershell
$env:MARIMO_SMOKE_ONLY=\"1\"
uv run python notebooks\\01_doubao_api_smoke.py
uv run python notebooks\\02_letta_e2e.py
Remove-Item Env:MARIMO_SMOKE_ONLY
```

6. If you want the notebook UI instead of script mode:

```powershell
uv run marimo run notebooks\\01_doubao_api_smoke.py --headless
uv run marimo run notebooks\\02_letta_e2e.py --headless
```

## What To Change If You Swap Models

If you change the Doubao model later:

Update both:
- `DOUBAO_CHAT_MODEL`
- `LETTA_MODEL_HANDLE`

Rule:
- `LETTA_MODEL_HANDLE` should match Letta's discovered handle format:
  - `openai-proxy/<exact-model-id>`

Recommended procedure:
1. Update `DOUBAO_CHAT_MODEL` in `.env`.
2. Update `LETTA_MODEL_HANDLE` to match.
3. Run direct smoke notebook first.
4. Then run the Letta e2e notebook.

## If You Want Pure Doubao Embeddings Later

This is the biggest unresolved area.

What is already known:
- The tested Ark compatibility path worked for chat and tool calling.
- The tested embedding requests did not work with the same key.

Possible next steps:
- Check whether Ark embeddings require different model IDs than the compatibility `/models` response implies.
- Check whether the account/key lacks access to the embedding models.
- Check whether Volcengine expects endpoint IDs or another naming layer for embeddings.
- If embeddings become available, update:
  - `.env`
  - the e2e notebook default embedding choice
  - the documentation

## If You Need External Exposure Later

Current state:
- No reverse proxy
- No TLS
- No extra auth layer beyond what Letta itself does

If the stack is exposed beyond local development:
- Add a reverse proxy
- Add TLS
- Decide on auth and network boundaries explicitly

Nginx was intentionally omitted from this bundle because it is not needed for local validation.

## If You Add More Services Later

This bundle is meant to be extended.

Good candidates:
- an app service that calls Letta
- a reverse proxy
- a monitoring service
- a worker service

Notes:
- `.dockerignore` is already in place for future build contexts.
- Keep the Letta and DB pieces isolated and stable before layering more services on top.

## Short Answers To Likely Future Questions

### Is `init.sql` still necessary?

Yes.

### Do we need `.dockerignore` right now?

Not for current behavior, because there is no Docker build in this bundle. It was added as future-proofing.

### Why not use the repo Dockerfile?

Because the official Letta image already works, and the goal was a portable standalone folder.

### Why not use Doubao embeddings?

Because the tested Ark key/path did not produce a working embedding request, while Letta's built-in embedding handle did.

### Why keep both `.env.example` and `.env`?

`.env.example` is the safe template. `.env` is the local runnable config.
