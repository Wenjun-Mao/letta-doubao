# Letta + Doubao Manual

This document is the durable handoff for the standalone bundle. It records the design decisions, what was verified, what failed, and what to keep in mind when moving this directory somewhere else.

## Goal

Create a standalone directory that can run Letta against Doubao without depending on the rest of the Letta source repository.

## Current State

The current bundle works end to end with:

- Letta server in Docker
- Postgres + pgvector in Docker
- Doubao Ark as the LLM backend through Letta's OpenAI-compatible provider path
- Letta's built-in embedding backend for agent memory
- Two marimo notebook files that can also be executed as script-based smoke tests

Current development baseline defaults:

- System prompt: `CHAT_V20260418_PROMPT`
- Test embedding: `letta/letta-free`
- Comment Lab task shape: `classic`
- Label Lab schema: `label_span_annotations_v1`

## Directory Contents

- `compose.yaml`: standalone Docker Compose stack
- `init.sql`: required database bootstrap
- `.env.example`: sanitized configuration template
- `.env`: local runtime configuration
- `pyproject.toml`: notebook dependency manifest for `uv`
- `uv.lock`: dependency lockfile generated by `uv`
- `README.md`: short run instructions
- `MANUAL.md`: this longer decision log and runbook
- `notebooks/01_doubao_api_smoke.py`: direct Ark validation
- `notebooks/02_letta_e2e.py`: Letta end-to-end validation

Maintained verification entry points under `tests/`:

- `pytest` suites under `tests/test_*.py`
- `tests/checks/platform_api_e2e_check.py`
- `tests/checks/ade_mvp_smoke_e2e_check.py`

General-purpose operational scripts remain under `scripts/`.

## Core Decisions

### 1. Standalone Directory Instead Of Reusing The Repo Root

Decision:
- Package everything under one directory so it can be copied elsewhere unchanged.

Why:
- The repo checkout is useful for investigation, but it should not be a runtime requirement.
- The final objective was future reuse after moving to a clean directory.

Consequence:
- No source build is required.
- The standalone directory is the artifact to keep.

### 2. Docker Compose Over Dockerfile

Decision:
- Use Docker Compose with the official `letta/letta:latest` image instead of building from the repo Dockerfile.

Why:
- The user explicitly preferred Compose.
- The official image is enough for this setup.
- Avoiding a build removes unnecessary moving parts.

Consequence:
- The standalone bundle does not need the repo Dockerfile.
- There is currently no Docker build context in this bundle.

### 3. Minimal Services

Decision:
- Keep only:
  - Letta server
  - Postgres + pgvector

Why:
- Nginx is not required for local validation.
- Fewer services make the bundle easier to move and debug.

Consequence:
- Letta API is exposed directly on `8283`.
- Letta ADE/UI is exposed directly on `8083`.

### 4. Doubao Through Letta's OpenAI-Compatible Path

Decision:
- Configure Letta via:
  - `OPENAI_API_BASE=https://ark.cn-beijing.volces.com/api/v3`
  - `OPENAI_API_KEY=<ark key>`

Why:
- Letta already supports OpenAI-compatible providers through this path.
- Ark supports compatible `/models` and `/chat/completions` behavior.

Consequence:
- Letta discovers Doubao-backed models as `openai-proxy/<model-id>`.
- The tested model handle is `openai-proxy/doubao-seed-1-8-251228`.

### 5. Doubao For LLM, Not For Embeddings

Decision:
- Use Doubao for the agent LLM.
- Use `letta/letta-free` for embeddings.

Why:
- The Ark key successfully supported:
  - `GET /models`
  - `POST /chat/completions`
  - OpenAI-style tool calling
- The tested embedding requests did not work with the same key on the tested compatibility path.

What failed:
- Alias-style embedding model name: `doubao-embedding`
- Tested explicit IDs returned by `/models`, such as:
  - `doubao-embedding-text-240715`
  - `doubao-embedding-large-text-240915`

Observed result:
- Ark returned `InvalidEndpointOrModel.NotFound` for those embedding requests with this key.

Consequence:
- The current working end-to-end stack is not "pure Doubao" for embeddings.
- If pure Doubao embeddings are required later, that needs a separate investigation.

### 6. Keep `init.sql`

Decision:
- Keep `init.sql` in the standalone bundle.

Why:
- It is still necessary for this database layout.
- It does all of the following:
  - connects to the target database
  - creates a dedicated schema named after the DB
  - sets the database `search_path`
  - creates the `vector` extension in that schema
  - drops the default `public` schema

Consequence:
- `init.sql` is a main file and should be preserved.
- Removing it changes the DB initialization behavior and would likely create avoidable startup/runtime issues.

### 7. Use `uv` Instead Of `pip`

Decision:
- Use `uv` as the dependency manager for the notebooks.

Why:
- The user explicitly requested it.
- The main Letta repo already uses `uv`.
- `uv sync` + `uv.lock` gives a cleaner portable workflow than `pip install -r requirements.txt`.

Consequence:
- `pyproject.toml` and `uv.lock` are the source of truth for Python dependencies.
- The old `requirements.txt` was removed to avoid two competing workflows.

### 8. Add `.dockerignore`, But Treat It As Future-Proofing

Decision:
- Add a minimal `.dockerignore`.

Why:
- It keeps CI Docker build context small for Agent Platform API image publishing.
- It avoids shipping local artifacts (for example `data/`, `.tmp/`, diagnostics outputs) into image builds.

Consequence:
- `.dockerignore` is part of active CI/CD performance and reproducibility.
- It reduces build time and accidental context bloat.

### 9. Option A Deployment Model (Pinned Letta + Prebuilt Agent Platform API)

Decision:
- Use pinned upstream Letta image for `letta_server`.
- Use prebuilt GHCR image for `agent_platform_api`.

Why:
- Keeps Letta maintenance burden low (no custom forked server image lifecycle).
- Delivers deterministic remote startup for the backend API without runtime dependency installs.
- Supports pull-first deployment on remote Ubuntu (`docker compose pull` + `up -d`).

Consequence:
- Image controls are environment-driven:
  - `LETTA_SERVER_IMAGE` (default `letta/letta:0.16.7`)
  - `AGENT_PLATFORM_API_IMAGE` (default `ghcr.io/wenjun-mao/agent-platform-api:latest`)
- CI publishes `agent_platform_api` via `.github/workflows/publish-agent-platform-api-ghcr.yml`.
- Remote operators should avoid `--build` for routine deploys.

## 2026-04 Debug Journey Notes

### Letta Startup Delay Root Cause (NLTK)

Observed:
- In restricted/no-egress environments, startup could stall at `Checking NLTK data availability...` while watchdog logs continued.

Root cause:
- Letta startup calls `nltk.download("punkt_tab")` during lifespan before readiness is achieved.
- Even with local NLTK data present, that download call may still attempt network access.

Implemented fix in this bundle:
- Pre-seeded local NLTK cache (`data/nltk_data`) via `scripts/seed_nltk_data.sh`.
- Mounted local cache into the Letta container.
- Added runtime patch at `docker/sitecustomize.py` to short-circuit `punkt_tab` download when local data exists, with strict-local mode and timeout fallback.

Validation:
- Internal no-egress simulation reached healthy state.
- In-network curl health probe succeeded.

### agent_platform_api Cold-Start Delay Root Cause

Observed:
- `agent_platform_api` startup logs showed runtime env creation and package download (for example `Downloading pydantic-core`), causing slow first response.

Root cause:
- `agent_platform_api` previously ran `uv run ...` directly in container startup, which performs dependency sync at runtime.

Implemented fix in this bundle:
- Added `docker/agent_platform_api.Dockerfile`.
- Moved dependency installation to image build (`uv sync --frozen --no-dev --no-install-project`).
- Published/pulled `agent_platform_api` as a prebuilt GHCR image in Option A while keeping Uvicorn runtime entrypoint from `/opt/venv/bin/uvicorn`.

Expected result:
- After build, `agent_platform_api` boots directly to Uvicorn without startup-time dependency download.

## Runtime Details That Matter

### Shared ADE Model Catalog

Important:
- `agent_platform_api` now uses `AGENT_PLATFORM_MODEL_SOURCES` as the shared source of truth for Agent Studio, Comment Lab, and Label Lab model discovery.

Notes:
- Each source entry declares the base URL, enabled scenarios, Letta handle prefix, and auth lookup fields.
- Auth resolution order is `/run/secrets/<api_key_secret>` first, then the env var named by `api_key_env`.
- Agent Studio only surfaces models whose derived Letta handles are actually registered in Letta.
- Comment Lab uses source-scoped model keys in the form `<source-id>::<provider-model-id>` so duplicate model names remain distinct across providers.
- The local port split is intentional:
  - `8081` is the active llama-server endpoint for local Comment Lab and Label Lab work.
  - `2234` remains optional Unsloth Studio standby.
  - `1234` remains optional LM Studio standby plus Letta bootstrap compatibility.
- llama-server runtime settings remain host-side operations. Loaded GGUF, context sizing, GPU offload, and reasoning mode are expected to be adjusted in the launch command on the machine running llama-server, not through ADE.
- Unsloth Studio and LM Studio may be offline or have no loaded model for long periods. In that state they should either be disabled in `AGENT_PLATFORM_MODEL_SOURCES` or appear only as `unreachable`/`empty` in diagnostics without blocking normal llama-server-first use.
- Ark model visibility is filtered through the checked-in allowlist report at `agent_platform_api/catalog_data/ark_chat_probe_report.json`, which is regenerated with `uv run python scripts/probe_provider_models.py --source-id ark --mode chat-probe --write`.
- Ark is currently enabled for chat/comment only. Label Lab is centered on llama-server JSON Schema output until a provider is explicitly enabled and verified for label schemas.

### Label Lab And Schema Center

Important:
- Label Lab is stateless like Comment Lab, but it produces structured span annotations instead of prose.
- Prompts define labeling criteria. Schemas define the JSON output contract.
- The default label schema is stored at `schemas/label/label_span_annotations_v1.json` and is surfaced through the Label Schema Center.

Notes:
- Schema Center v1 is label-only. It provides create, edit, archive, restore, and purge operations under `/api/v1/platform/schema-center/label-schemas`.
- The API container mounts `./schemas:/app/schemas`, so UI edits persist to the host filesystem like Prompt Center edits.
- `POST /api/v1/labeling/generate` requires a selected `model_key`, `prompt_key`, and `schema_key`.
- For the `llama_cpp_server` adapter, Label Lab sends `response_format={"type":"json_schema","json_schema":...}` to `/v1/chat/completions`, matching the notebook-proven llama-server pattern.
- Server-side validation still verifies the parsed shape and exact Unicode character offsets, including `input[start:end] == text` and no overlapping spans.
- If a provider returns the right unique `text` with wrong offsets, ADE normalizes the offsets before accepting the result; ambiguous or missing text still fails validation.

### Ark Usability Allowlist

Important:
- Ark's raw `/models` response is broader than the set of models that actually answer a minimal text `/chat/completions` request.
- ADE therefore uses a persisted allowlist generated by the probe script instead of trusting raw discovery for UI visibility.

Current seeded usable set:
- `doubao-1-5-lite-32k-250115`
- `doubao-seed-1-6-250615`
- `doubao-seed-1-6-vision-250815`
- `doubao-seed-1-6-251015`
- `glm-4-7-251222`
- `doubao-seed-1-8-251228`
- `doubao-seed-2-0-lite-260215`
- `doubao-seed-2-0-mini-260215`
- `doubao-seed-2-0-pro-260215`

### Model Handle Naming

Important:
- In Letta, the Doubao model handle is not `openai/doubao-seed-1-8-251228`.
- It is `openai-proxy/doubao-seed-1-8-251228`.

Reason:
- Letta uses `openai-proxy` for non-OpenAI providers routed through the OpenAI-compatible provider implementation.

### Full Model ID Matters

Important:
- Use the actual Ark model ID, not the display name.

Working:
- `doubao-seed-1-8-251228`

Do not assume:
- `doubao-seed-1-8` alone will behave the same way in Letta's discovered model list.

### Letta Model Routes Redirect

Observed:
- Letta's model listing route redirects to a trailing-slash URL.

Example:
- `GET /v1/models` redirected to `GET /v1/models/`

Consequence:
- The e2e notebook was updated to use `follow_redirects=True` and `/v1/models/`.

### Marimo Notebook Files Also Run As Scripts

Decision:
- The notebook files support both:
  - normal marimo UI usage
  - script-style smoke execution

Mechanism:
- Set `MARIMO_SMOKE_ONLY=1` and run the notebook with Python.

Why:
- This makes smoke tests automatable without opening the notebook UI.

## Fresh Start Runbook

If you move this directory somewhere else and want a clean start:

1. Copy the entire directory.
2. Keep or recreate `.env`.
3. If you want a truly fresh DB, remove `data/pgdata`.
4. Run:

```powershell
uv sync
docker compose up -d
```

5. Run the smoke notebooks:

```powershell
$env:MARIMO_SMOKE_ONLY=\"1\"
uv run python notebooks\\01_doubao_api_smoke.py
uv run python notebooks\\02_letta_e2e.py
Remove-Item Env:MARIMO_SMOKE_ONLY
```

6. Run development verification:

```powershell
$env:PYTHONPATH="."
uv run python -m pytest
uv run python tests/checks/platform_api_e2e_check.py
uv run python tests/checks/ade_mvp_smoke_e2e_check.py
```

7. If you launch checks from Test Center, inspect transient orchestrator logs under:

- `tests/outputs/platform_orchestrator/`

8. If you want the notebook UI instead of script mode:

```powershell
uv run marimo run notebooks\\01_doubao_api_smoke.py --headless
uv run marimo run notebooks\\02_letta_e2e.py --headless
```

## What To Change If You Swap Models

If you change the Doubao model later:

Update both:
- `DOUBAO_CHAT_MODEL`
- `LETTA_MODEL_HANDLE`

Rule:
- `LETTA_MODEL_HANDLE` should match Letta's discovered handle format:
  - `openai-proxy/<exact-model-id>`

Recommended procedure:
1. Update `DOUBAO_CHAT_MODEL` in `.env`.
2. Update `LETTA_MODEL_HANDLE` to match.
3. Run direct smoke notebook first.
4. Then run the Letta e2e notebook.

If you change ADE provider discovery later:

Update:
- `AGENT_PLATFORM_MODEL_SOURCES`
- any referenced source auth env vars such as `LLAMA_SERVER_API_KEY`, `UNSLOTH_API_KEY`, or `OPENAI_API_KEY`

## If You Want Pure Doubao Embeddings Later

This is the biggest unresolved area.

What is already known:
- The tested Ark compatibility path worked for chat and tool calling.
- The tested embedding requests did not work with the same key.

Possible next steps:
- Check whether Ark embeddings require different model IDs than the compatibility `/models` response implies.
- Check whether the account/key lacks access to the embedding models.
- Check whether Volcengine expects endpoint IDs or another naming layer for embeddings.
- If embeddings become available, update:
  - `.env`
  - the e2e notebook default embedding choice
  - the documentation

## If You Need External Exposure Later

Current state:
- No reverse proxy
- No TLS
- No extra auth layer beyond what Letta itself does

If the stack is exposed beyond local development:
- Add a reverse proxy
- Add TLS
- Decide on auth and network boundaries explicitly

Nginx was intentionally omitted from this bundle because it is not needed for local validation.

## Short Answers To Likely Future Questions

### Is `init.sql` still necessary?

Yes.

### Do we need `.dockerignore` right now?

Not for current behavior, because there is no Docker build in this bundle. It was added as future-proofing.

### Why not use the repo Dockerfile?

Because the official Letta image already works, and the goal was a portable standalone folder.

### Why not use Doubao embeddings?

Because the tested Ark key/path did not produce a working embedding request, while Letta's built-in embedding handle did.

### Why keep both `.env.example` and `.env`?

`.env.example` is the safe template. `.env` is the local runnable config.
